# Git Push Scheduler

Use this skill to schedule staggered `git push` operations for commits that are already present locally on the current branch, then return an exact Bash or PowerShell script the user can run themselves.

This skill only pushes existing commits. It does not create commits, amend commits, rewrite authors, or rewrite commit messages. Treat the existing git history as the source of truth. If the audit finds AI-related markers in the commit metadata, stop and report them. Do not try to conceal or rewrite them.

## Workflow

1. Build the scheduler binary:

```bash
CARGO_TARGET_DIR=/tmp/git-push-scheduler-target \
  cargo build --locked --release --manifest-path /Users/iwan/.codex/skills/git-push-scheduler/scripts/push_scheduler/Cargo.toml
```

2. Create a plan file for the target repository:

```bash
/tmp/git-push-scheduler-target/release/push_scheduler \
  plan \
  --repo /absolute/path/to/repo \
  --plan-file /tmp/git-push-scheduler.plan
```

3. Read the planner output and summarize it back to the user before any push starts. Always include:
   - the branch and tracked upstream branch
   - the distinct author and committer identities already present in the pending commits
   - the delay before each push and the cumulative time offset
   - the total runtime from first push to last push
   - any audit failures or safety warnings

4. Wait for explicit user approval before generating the runner script.

5. Generate the exact saved runner script only after approval:

```bash
/tmp/git-push-scheduler-target/release/push_scheduler \
  script \
  --plan-file /tmp/git-push-scheduler.plan \
  --shell bash
```

6. Return the generated script to the user in a fenced code block so they can copy-paste and run it themselves. If they ask for Windows, generate `--shell powershell` instead.

## Operating Rules

- Work only with commits that are already ahead of `@{upstream}`. This skill is for committed work, not staged-but-uncommitted changes.
- Require a tracked upstream branch. If the current branch has no upstream, stop and ask the user to configure one or choose a different repository.
- Preserve the existing commit metadata. `git push` does not change author or message fields, and this skill must not attempt to do so.
- Build with `CARGO_TARGET_DIR=/tmp/git-push-scheduler-target` and `--locked` so the crate can compile cleanly without writing build artifacts into the global skills directory.
- If the remote branch moves after the plan is created, the generated script must fail closed and ask the user to re-plan instead of forcing the schedule through.
- If additional local commits appear after the plan is created, the generated script must fail closed and ask the user to re-plan so the published schedule remains accurate.

## Commands

### `plan`

Use `plan` to inspect the local commits ahead of upstream, classify each commit by change size, generate irregular delays, print the schedule, and save the exact plan to disk.

Supported options:
- `--repo <path>`: Repository root or any path inside the repository. Defaults to the current directory.
- `--plan-file <path>`: Output plan file path. Defaults to `/tmp/git-push-scheduler.plan`.
- `--min-delay-minutes <n>`: Lower clamp for generated delays after the first push. Defaults to `12`.
- `--max-delay-minutes <n>`: Upper clamp for generated delays after the first push. Defaults to `90`.
- `--seed <n>`: Optional seed for reproducible plans.

### `script`

Use `script` to read a previously saved plan file and emit a copy-paste Bash or PowerShell script that performs the scheduled pushes with branch and remote-drift checks.

Supported options:
- `--plan-file <path>`: Path to the plan file generated by `plan`.
- `--shell <bash|powershell>`: Output shell syntax. Defaults to `bash`.
- `--output <path>`: Optional output file path. Without it, the script is printed to stdout.

### `execute`

Use `execute` to read a previously saved plan file and push each commit SHA directly to the tracked remote branch at the scheduled offsets. This is available for direct local use, but the default skill workflow should prefer `script` so the user explicitly runs the schedule.

Supported options:
- `--plan-file <path>`: Path to the plan file generated by `plan`.
- `--yes`: Required confirmation flag. Without it, execution must fail closed.

## Failure Cases

- Audit failure:
  If commit author, committer, subject, or body contains terms such as `codex`, `copilot`, `claude`, `chatgpt`, `gpt`, `openai`, `anthropic`, `llm`, `ai-generated`, or similar markers, stop and show the user which commit and field triggered the audit.

- Remote drift:
  If the tracked remote branch no longer points to the expected commit before any scheduled push, stop and ask the user whether to re-plan.

- Branch mismatch:
  If the current branch is not the same branch the plan was created for, stop and ask the user whether they intended to switch branches.

- No pending commits:
  If `@{upstream}..HEAD` is empty, stop and tell the user there is nothing to schedule.

## Example Invocation

Use when the user asks for something like:
- `Use $git-push-scheduler to spread the five local commits on this branch over a realistic schedule.`
- `Use $git-push-scheduler to show me a push plan for the commits ahead of origin/main, then return the Bash script after I approve it.`
